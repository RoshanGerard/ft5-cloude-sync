## Why

The desktop file explorer's `window.api.files.*` IPC surface still routes `list` / `stat` / `search` / `remove` through the `mock-fs` module in `apps/desktop/src/main/ipc/files/`. This was a stub during the explorer UI build-out and has outlived its purpose: every user connects a real Google Drive, OneDrive, or S3 datasource but sees the same fake entries. The `@ft5/fs-datasource-engine` package already exposes the exact methods the explorer needs (`listDirectory`, `getMetadata`, `search`, `deleteFile`, `deleteDirectory`) and is proven in production by the `fs-sync-service`, so this change closes the loop between the explorer UI and the live engine without any new engine capability. The explorer also currently has no distinct treatment for datasources that are disconnected, auth-revoked, initial-syncing, or connected-but-empty — users see plaintext "Failed to load" or nothing — and the UX rules in `CLAUDE.md` require an approved visual direction before implementing these states.

## What Changes

- **BREAKING** Add four new commands to the `fs-sync` service RPC surface (`packages/ipc-contracts/src/sync-service/commands.ts` `CommandMap`): `files:list`, `files:stat`, `files:search`, `files:remove`. The sync service hosts the engine; the desktop main process reaches the engine through these new RPCs, not by instantiating a second engine client in main.
- Implement the four new command handlers inside `services/fs-sync` by delegating to `client.listDirectory` / `client.getMetadata` / `client.search` / `client.deleteFile` (or `client.deleteDirectory`).
- Replace `mock-fs` delegation in the desktop main-process IPC handlers `apps/desktop/src/main/ipc/files/{list,stat,search,remove}.ts` with calls to the sync-service RPC (`SyncClient.request("files:list", …)`).
- Carry a discriminated-union error envelope from the engine through the sync-service RPC through the main IPC response, so the renderer can distinguish `"auth-revoked"`, `"disconnected"`, `"rate-limited"`, and `"other"` deterministically instead of string-matching an opaque `message` field. Update `window-api.d.ts` and preload contracts accordingly.
- Introduce per-state UI treatments in the renderer's file-explorer view: `loading` (skeleton rows matched to the active view mode), `disconnected` (pattern A full-replace with Retry), `auth-revoked` (pattern A full-replace with Reconnect), `syncing` (pattern A full-replace with spinner + progress label), `connected-but-empty` (pattern A full-replace, neutral tone). Visual direction approved during `/opsx:propose` is recorded in `design.md` under "Visual direction".
- Disable the Rename and Download affordances (toolbar button + context-menu items) in the file-explorer UI whenever the datasource is engine-backed (i.e. not a synthetic mock datasource). A tooltip "Coming soon — see change `add-engine-rename-download`" explains the disabled state. Prevents users from silently mutating mock state while the rest of the explorer shows real files.
- Treat `paused` datasources identically to `connected` for browsing: sync being off does not prevent listing or navigation. No additional UI state is surfaced for `paused`.
- Engine response is authoritative when it conflicts with the datasources-store status: if the store reports `connected` but `listDirectory` rejects with `tag: "auth-revoked"`, the auth-revoked state is shown. The store is treated as a predictive/optimistic hint for initial render only.
- Bulk removal (UI passes `string[]`) is best-effort: each path is attempted via `deleteFile` or `deleteDirectory` and results are aggregated with a per-path success/failure list; the renderer surfaces an aggregate toast (e.g., "3 of 5 deleted; 2 failed"). Engine's per-target semantics are preserved — no transactional rollback.
- Preserve the renderer's request-ordering guarantee (`requestIdRef` stale-response guard in `useExplorerData`) and every existing test for the view-modes, store, breadcrumb, toolbar, and status row.
- Keep `files.rename` and `files.download` on `mock-fs` for now. These two IPC surfaces require engine capabilities that do not yet exist and are explicitly scoped to a sibling change `add-engine-rename-download`. `mock-fs.ts` stays in the tree until that change lands.

### Modified Capabilities
- `file-explorer`: the `window.api.files.*` IPC surface's data source shifts from `mock-fs` to the live engine (via `fs-sync-service` RPC) for `list`, `stat`, `search`, and `remove`. New requirements cover the five surfaced states (loading skeleton, disconnected, auth-revoked, syncing, connected-but-empty) with pattern A full-replace visuals and Lucide iconography. The existing "300-entry directory ceiling enforced on mocked fixtures" requirement is dropped. Rename and Download affordances become conditionally disabled in the UI but their IPC handler semantics are unchanged in this delta.
- `fs-sync-service`: CommandMap gains `files:list`, `files:stat`, `files:search`, `files:remove` with typed params/results/errors, each delegating to the engine client held by the service. This is a **BREAKING** shape change to the `CommandMap` surface; any consumer that enumerates `COMMAND_NAMES` will see four additional entries.

## Impact

- **IPC contracts (`packages/ipc-contracts/src/sync-service/commands.ts`)** — CommandMap extended with four new commands and their param/result/error shapes. `COMMAND_NAMES` tuple updated. `packages/ipc-contracts/src/files.ts` response types widened to carry the discriminated-error envelope.
- **fs-sync service (`services/fs-sync/src/commands/`)** — four new command handlers implemented, each resolving the engine client by `datasourceId` and calling `listDirectory` / `getMetadata` / `search` / `deleteFile|deleteDirectory`; errors normalized into the envelope.
- **Main process (`apps/desktop/src/main/ipc/files/`)** — `list.ts`, `stat.ts`, `search.ts`, `remove.ts` rewritten to `syncClient.request("files:list", …)` etc. `mock-fs.ts` remains imported only by `rename.ts` and `download.ts` until the sibling change.
- **Renderer (`apps/desktop/src/renderer/src/features/file-explorer/`)** — `file-explorer.tsx` gains branching for the five states; new state components under `file-explorer/states/` (`disconnected.tsx`, `auth-revoked.tsx`, `syncing.tsx`, `empty.tsx`, `skeleton.tsx`); `use-explorer-data.ts` maps engine error tags to states and subscribes to datasources-store status for predictive render; `toolbar.tsx` and the per-entry context menu honor the disable-when-engine-backed rule for Rename and Download.
- **Dependencies** — none added. `@ft5/fs-datasource-engine` stays internal to the sync service. Icons from the existing `lucide-react` adapter (`components/icon.tsx`).
- **Known limitations (out of scope, tracked)** — `listDirectory` returns at most one provider page (Drive `pageSize: 1000`, OneDrive default, S3 list limit). Folders exceeding that silently truncate. Flagged in `design.md` under "Known limitations" with a proposed follow-up change name `add-engine-listdirectory-pagination`. Not addressed here.
- **Out of scope (explicit)** — `files.rename`, `files.download`, packaged-build runtime (defect #8 from `wire-fs-sync-service`), OAuth UI, directory pagination.
