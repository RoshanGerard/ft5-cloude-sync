## Why

The app's job surface — upload a local file to a cloud datasource, keep a local tree mirrored to a cloud root — cannot reliably live inside the Electron main process. When the user closes the desktop app, the main process exits and any in-flight job dies with it. The existing `services/fs-monitor` watches local files but has no executor; the `@ft5/fs-datasource-engine` knows how to talk to providers but has no queue, persistence, retry, or lifecycle of its own. Every in-flight product conversation (manual sync, auto-sync-on-change, retry-when-network-returns) assumes a background process that owns jobs independently from the app. This change introduces that process: `services/fs-sync` — a per-user OS service that accepts manual upload and one-way mirror-sync jobs, persists them to SQLite, executes them against the engine with a global concurrency cap, and keeps running after the desktop app closes.

## What Changes

- Introduce a new workspace service `services/fs-sync/` — a framework-agnostic Node.js TypeScript daemon. Consumes `@ft5/fs-datasource-engine` and `@ft5/ipc-contracts`. Does not import Electron.
- Run the service as a **per-user OS service** (Windows Scheduled Task at login / macOS LaunchAgent / Linux `systemd --user` unit) — NOT as a system service. Watches and acts on the current user's own files only.
- Ship a named-pipe JSON-line IPC transport: `\\.\pipe\ft5-sync` on Windows, `$HOME/ft5/sync_app/sync.sock` on Unix. One bidirectional channel per connected client; server pushes unsolicited event frames alongside command responses.
- Add a new `@ft5/sync-service-contracts` package (or equivalent additions to `@ft5/ipc-contracts` — choice documented in `design.md`) carrying: the IPC framing types, the command surface (`sync:enqueue-upload`, `sync:enqueue-mirror`, `sync:list-jobs`, `sync:cancel-job`, `sync:get-job`, `sync:subscribe-events`, `sync:unsubscribe-events`, `sync:set-retry-policy`, `sync:authenticate`), the event surface, and the `SyncAlreadyRunningError` shape.
- Introduce a new `ConfigFileCredentialStore` implementation of the engine's `CredentialStore` port at `services/fs-sync/src/credential-store/config-file.ts`. Reads/writes plaintext JSON at `$HOME/ft5/sync_app/credentials.json` with `0600` / user-only Windows ACL. Refuses to operate if the file's permission mask is broader than user-only on Unix. Plaintext is a deliberate deferred-security posture documented in `design.md`; OS-keyring migration is out of scope for this change.
- Persist jobs and a local-to-cloud sync snapshot in SQLite at `$HOME/ft5/sync_app/sync.db` using Drizzle ORM. Schema is owned by the sync service; the desktop app does NOT connect to this database. Drizzle import grep-allowlist is extended to include `services/fs-sync/src/`.
- Implement a `JobScheduler` enforcing a global concurrency cap of **2** concurrent jobs via a semaphore, with a deterministic FIFO queue for the rest. Scheduler is pluggable: sequential-only fallback MUST be selectable by construction-time policy for operability.
- Implement the `Job` abstraction as a Strategy/Command pattern: `UploadJobExecutor` for manual single-file uploads; `MirrorSyncJobExecutor` for one-way local-to-cloud mirror-sync. Both executors consume a `DatasourceClient<T>` from the engine's `ClientFactory`. Conflict handling is driven by a per-job `conflictPolicy: 'overwrite' | 'duplicate' | 'skip'` set at enqueue time — the service NEVER prompts the user mid-job.
- Implement snapshot-driven mirror-sync semantics: walk the local source tree, diff against a persisted `sync_snapshot` table keyed by `(datasourceId, relPath)`, upload or re-upload entries that differ by `(size, mtime)` then `sha256`, and delete remote entries whose local counterpart no longer exists. There is **no** mass-delete threshold. A **source-health precondition** SHALL be enforced: if the source root is missing, unreadable, or empty under a read error, the job pauses with a `source-unavailable` event and NO remote deletes are issued.
- Implement a **dedup rule** for mirror-sync: if a `sync` job already exists in `queued` or `running` status for the same `(datasourceId, sourcePath)`, reject the new `sync:enqueue-mirror` command with a `SyncAlreadyRunningError` response carrying the existing job id, the datasource id, and the source path. The dedup rule does NOT apply to `upload` jobs.
- Implement a **retry split**. System-level retries (non-configurable, engine-guided): `DatasourceError.tag === 'network-error'` → move job to `waiting-network` state; `rate-limited` → honour `retryAfterMs`; `auth-expired` → engine's single-flight refresh already handles it. User-level retries (configurable per datasource or global via `sync:set-retry-policy`) apply only to `provider-error` and unclassified failures, and only when the engine's `DatasourceError.retryable === true`. `auth-revoked`, `not-found`, `conflict`, and `unsupported` are terminal.
- Implement **error-driven network monitoring**: a single background probe polls `dns.resolve` against a well-known host every 30 seconds while at least one job is in `waiting-network` state; on success, all `waiting-network` jobs transition back to `queued`. The probe is idle (and consumes no cycles) when no jobs are waiting. No native modules are introduced; no OS network-change hooks are used.
- **Full re-upload** on network retry. Resumable-upload session APIs (S3 multipart, Drive resumable sessions, OneDrive upload sessions) are NOT implemented in this change; a partial upload that fails is re-started from byte 0 on retry.
- Declare a `MonitorEventSource` input port for future auto-sync. Ship a **no-op default implementation** in v1; the port accepts no events until `services/fs-monitor` is wired in a follow-up change. Auto-sync command surface is NOT exposed in v1 — the desktop app can only trigger manual sync.
- Provide a **dev mode** via a new pnpm workspace script that spawns the service as a detached Node process on the developer's machine with a distinct dev-only pipe name (`\\.\pipe\ft5-sync-dev` / `$HOME/ft5/sync_app/sync-dev.sock`) and a distinct dev data dir, so a developer can run the desktop app against a local service without touching their production-installed service.
- Provide a **prod install hook** via electron-builder: the installer registers the per-user service on post-install and deregisters on uninstall. Registration uses OS-native tools (`schtasks` on Windows, `launchctl` on macOS, `systemctl --user` on Linux) — no third-party service-manager library is introduced. Exact wiring is documented in `design.md`.

## Capabilities

### New Capabilities

- `fs-sync-service`: the background daemon's public contract — its per-user process model and lifecycle; the named-pipe IPC channel shape and framing; the command and event vocabulary; the `Job` state machine; global concurrency and dedup semantics; the one-way mirror-sync algorithm including source-health precondition and deletion propagation; the snapshot schema and diff algorithm; the retry split and error-driven network probe; the `ConfigFileCredentialStore` implementation of the engine's `CredentialStore` port (including file-permission enforcement); the `MonitorEventSource` input port; and the dev-mode / prod-install hooks.

### Modified Capabilities

None. The engine's `CredentialStore` port is a stable abstract surface; adding a second implementation does not change the port's spec. `datasources-ui` is not touched because desktop-app wiring (both command issuance and event subscription) is explicitly out of scope for this change.

## Impact

- **New workspace service.** `services/fs-sync/` — framework-agnostic Node.js 24.14.1 LTS TypeScript 6.0.3 (strict). Imports: `@ft5/fs-datasource-engine`, `@ft5/ipc-contracts`, Drizzle ORM, `better-sqlite3`, Node built-ins (`net`, `fs`, `crypto`, `dns/promises`). Added to the pnpm workspace.
- **New runtime dependencies** (to be justified in `design.md`): `drizzle-orm`, `better-sqlite3` (same stack already used by `apps/desktop`). No new native modules, no new IPC-framework dependencies — Node's built-in `net` handles both Windows named pipes and Unix domain sockets uniformly.
- **Contract surface expands.** `@ft5/ipc-contracts` (or a new sibling package — decision in `design.md`) gains: sync-service command union, response union, event union, `SyncAlreadyRunningError` shape, retry-policy shape, `MonitorEventSource` port types, and the pipe-name/socket-path convention constants. All additive; nothing removed.
- **Credential storage.** A new `credentials.json` file at `$HOME/ft5/sync_app/credentials.json` becomes the single source of truth for datasource credentials consumed by the service. The file is plaintext JSON with `0600` / user-ACL permissions. The desktop app's future integration will read/write this same file so the two processes never diverge on credentials. This supersedes, for service-consumed credentials, the engine's `SqliteCredentialStore` / `safeStorage` path; that path remains valid for any in-process Electron engine use but is NOT consulted by the service.
- **SQLite database.** A new service-owned `sync.db` at `$HOME/ft5/sync_app/sync.db` with tables `jobs`, `sync_snapshot`, `retry_policies`, and `service_meta`. Migrations owned by the service. The desktop app MUST NOT open this database; all access goes through the IPC surface.
- **Installer.** `apps/desktop`'s electron-builder config gains post-install and post-uninstall hooks that register/deregister the per-user service via OS-native commands. No installer plugin is introduced.
- **Workspace scripts.** `package.json` (root) gains `dev:sync-service` to spawn the service in detached mode for local development, and `build:sync-service` to produce the distributable binary/bundle packaged by the installer.
- **Drizzle allowlist.** The CI grep that enforces "Drizzle imports only under `apps/desktop/src/main/`" is extended to also permit imports under `services/fs-sync/src/` and `services/fs-monitor/src/` (the latter is already implicitly required by `openspec/project.md`'s stack description).
- **Cross-change.** `services/fs-monitor` (future change) will supply a real `MonitorEventSource` implementation and drive the currently-dormant auto-sync path. Desktop-app integration (calling the sync-service IPC surface, subscribing to its events, surfacing job status in the renderer) is a separate follow-up change. Resumable-upload support is a follow-up engine change. OS-keyring credential storage is a follow-up change.
- **Testing.** The service ships with contract tests against an in-memory `FakeDatasourceClient` (reused from the engine's fixtures) so every job state transition, dedup rule, retry path, and source-health branch is verified without real-provider network calls. Real-provider integration tests remain out of scope here.
- **Security.** Plaintext credentials on disk is a known limitation documented as a deferred tradeoff in `design.md` with a migration plan to OS keyring (`keytar` / `wincred` / libsecret) in a future change. Mitigations in v1: strict file permissions (0600 / user-ACL), per-user service identity (credentials never accessible to other OS users on a shared machine), refusal-to-operate if file permissions widen.
