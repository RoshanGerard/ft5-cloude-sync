### 5.B DEFERRED to follow-up change

Section 5.B was drafted to add a no-creds engine construction path (`factory.createForAuth`) so the service-side handlers could call `client.authenticate()` without pre-existing credentials. Drafting the first subagent surfaced a deeper architecture gap: OAuth strategies (`googledrive-client.ts:636`, `onedrive-client.ts:466`) read `this.creds.{clientId, clientSecret, redirectUri, tenantId}` during `doAuthenticateImpl` to build the `authorizeUrl` and inside the `completeWith` closure. OAuth *app* config is not currently sourced from anywhere except `StoredCredentials.authResult.meta`; there is no service-level config, no env source, no UI flow that collects it.

Fixing this properly requires: (a) the engine adjustment originally planned as 5.B, plus (b) a new service-level config sourcing mechanism for OAuth app config, plus (c) a renderer onboarding UI that collects it. That is a standalone feature and will ship in a follow-up change (`implement-datasource-onboarding`). See `design.md` Decision 11.

Section 5.A service handlers (5.A.7–5.A.10) therefore ship as **stubs** returning `{ ok: false, error: { tag: "not-implemented" } }`. The correlation store (5.A.5/5.A.6) stays in the codebase as dormant infrastructure for the follow-up. The wire contract, desktop-side wrappers, IPC handlers, and preload surface all ship with the final two-command shape; the `not-implemented` error propagates through them.
